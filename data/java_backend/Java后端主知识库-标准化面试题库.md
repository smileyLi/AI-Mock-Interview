## Q001 Java 对象比较

### 元数据
- 岗位：Java后端开发工程师
- 题型：技术知识
- 主题：Java基础
- 难度：初级
- 关键词：`equals` `==` `hashCode` `对象比较`

### 面试题
请你解释一下 `==` 和 `equals()` 的区别？`hashCode()` 和它们之间有什么关系？

### 标准回答
`==` 比较的是两个变量是否指向同一份值。对于基本类型，它比较的是具体数值；对于引用类型，它比较的是两个引用是否指向同一个对象。`equals()` 是 `Object` 的方法，默认实现本质上也是比较引用地址，但很多类会重写它，例如 `String` 会改成比较内容是否相等。`hashCode()` 通常和 `equals()` 一起使用，核心约束是：如果两个对象 `equals()` 为 `true`，那么它们的 `hashCode()` 必须相同；反过来 `hashCode()` 相同，并不代表 `equals()` 一定相同。这个约束对 `HashMap`、`HashSet` 这类哈希容器非常重要。

### 追问点
- 为什么重写 `equals()` 时通常也要重写 `hashCode()`？
- 如果只重写 `equals()` 不重写 `hashCode()`，在 `HashMap` 中会出现什么问题？
- 为什么 `String` 常被用作 `HashMap` 的 key？

### 评分点
- 能准确区分基本类型和引用类型下 `==` 的语义
- 能说明 `equals()` 的默认行为和重写后的意义
- 能说清楚 `hashCode()` 与哈希容器的关系

### 常见失分点
- 说 `==` 永远比较地址
- 说 `hashCode()` 相同就一定是同一个对象

## Q002 List 选型

### 元数据
- 岗位：Java后端开发工程师
- 题型：技术知识
- 主题：集合
- 难度：中级
- 关键词：`ArrayList` `LinkedList` `随机访问` `插入删除`

### 面试题
`ArrayList` 和 `LinkedList` 有什么区别？分别适合什么场景？

### 标准回答
`ArrayList` 底层是动态数组，支持通过下标快速访问，读取性能通常更好，但中间位置插入和删除元素时可能需要移动大量数据。`LinkedList` 底层是双向链表，理论上插入和删除节点更灵活，但随机访问性能较差，而且节点对象额外维护前驱后继指针，内存开销更高。实际工程里，如果主要需求是遍历和按索引访问，通常优先选 `ArrayList`；如果频繁在链表头尾进行插入删除，或者明确要使用队列/双端队列语义，才会考虑 `LinkedList`。单从大多数业务开发看，`ArrayList` 使用频率更高。

### 追问点
- 为什么 `LinkedList` 理论插入快，但很多真实业务里仍不如 `ArrayList` 常用？
- `ArrayList` 的扩容机制是怎样的？
- `ArrayList` 和普通数组的区别是什么？

### 评分点
- 能讲清楚底层数据结构差异
- 能结合访问模式说明选型依据
- 能意识到理论复杂度不等于真实工程性能

### 常见失分点
- 简化成“插入删除一定是 `LinkedList` 快”
- 不考虑 CPU 缓存友好性和遍历成本

## Q003 HashMap 底层原理

### 元数据
- 岗位：Java后端开发工程师
- 题型：技术知识
- 主题：集合
- 难度：中级
- 关键词：`HashMap` `哈希冲突` `扩容` `红黑树`

### 面试题
请你讲一下 `HashMap` 的底层实现原理。

### 标准回答
`HashMap` 是基于数组、链表和红黑树实现的哈希表。插入元素时，先根据 key 的 hash 计算桶下标，如果该桶为空就直接插入；如果发生哈希冲突，就在该桶内继续比较 key 是否相同，相同则覆盖，不同则挂到链表或红黑树中。JDK 8 之后，当链表长度超过树化阈值且数组容量足够时，链表会转换成红黑树，以提升极端冲突场景下的查询性能。`HashMap` 还会根据负载因子进行扩容，扩容后元素会重新分布到新的桶中。它的线程安全性较差，并发场景通常应使用 `ConcurrentHashMap` 或额外同步控制。

### 追问点
- 为什么数组长度通常设计为 2 的幂？
- `HashMap` 在 JDK 8 做了哪些变化？
- `HashMap` 为什么线程不安全？

### 评分点
- 能讲出数组加链表加红黑树的结构
- 能说明冲突处理和扩容机制
- 能提到 JDK 8 的树化优化

### 常见失分点
- 只会背“数组加链表”，忽略红黑树
- 把 `HashMap` 说成线程安全容器

## Q004 volatile 与线程安全

### 元数据
- 岗位：Java后端开发工程师
- 题型：技术知识
- 主题：并发
- 难度：中高级
- 关键词：`volatile` `可见性` `有序性` `原子性`

### 面试题
`volatile` 关键字有什么作用？它能保证线程安全吗？

### 标准回答
`volatile` 主要保证共享变量的可见性，并在一定程度上禁止指令重排序。一个线程修改了 `volatile` 变量后，其他线程能够更快看到最新值，因此它适合做状态标记、开关变量、双重检查单例中的实例引用修饰等场景。但 `volatile` 不能保证复合操作的原子性，例如 `i++` 仍然不是线程安全的，因为它包含读、改、写多个步骤。如果涉及计数累加、条件判断后更新等复合操作，通常还需要 `synchronized`、`Lock` 或原子类来保证线程安全。

### 追问点
- `volatile` 和 `synchronized` 的区别是什么？
- 为什么双重检查单例要配合 `volatile`？
- 为什么 `i++` 不是原子操作？

### 评分点
- 能说清可见性和有序性
- 能明确指出不保证复合操作原子性
- 能给出真实使用场景

### 常见失分点
- 说 `volatile` 可以完全替代锁
- 说 `volatile` 能保证所有线程安全问题

## Q005 线程池配置

### 元数据
- 岗位：Java后端开发工程师
- 题型：技术知识
- 主题：并发
- 难度：中级
- 关键词：`线程池` `ThreadPoolExecutor` `拒绝策略` `队列`

### 面试题
为什么生产环境里通常不建议直接使用 `Executors` 创建线程池？你一般会怎么配置线程池？

### 标准回答
不建议直接使用 `Executors` 的几个快捷工厂方法，主要是因为它们可能隐藏了不合理的默认参数。例如固定线程池可能配合无界队列，导致任务堆积和内存风险；缓存线程池则可能创建过多线程，造成系统资源耗尽。生产环境更推荐直接使用 `ThreadPoolExecutor`，显式指定核心线程数、最大线程数、阻塞队列、线程工厂和拒绝策略。配置时要结合业务类型判断：CPU 密集型任务更关注线程数不要超过 CPU 核数太多，IO 密集型任务则可以适当放大线程数；同时要监控队列长度、任务耗时、拒绝次数等指标，避免只配置不观测。

### 追问点
- 线程池处理任务的执行流程是什么？
- 常见拒绝策略有哪些，分别适合什么场景？
- 线程池里的异常应该怎么处理？

### 评分点
- 能指出 `Executors` 的潜在风险
- 能说出线程池核心参数的作用
- 能结合业务谈配置思路和监控思路

### 常见失分点
- 只背线程池参数名，不会解释实际意义
- 说线程池配置可以完全套模板，不需要结合业务

## Q006 JVM 运行时内存区域

### 元数据
- 岗位：Java后端开发工程师
- 题型：技术知识
- 主题：JVM
- 难度：中高级
- 关键词：`JVM` `内存区域` `堆` `栈` `方法区`

### 面试题
请你讲一下 JVM 运行时内存区域。

### 标准回答
JVM 运行时内存区域通常包括程序计数器、虚拟机栈、本地方法栈、堆和方法区。程序计数器是线程私有的，用来记录当前线程执行到哪条字节码指令；虚拟机栈保存方法调用过程中的栈帧信息；本地方法栈服务于 native 方法；堆是对象实例和数组分配的主要区域，也是垃圾回收的重点；方法区用于存储类元信息、常量、静态变量等。面试回答时最好把概念和排障联系起来，例如频繁 Full GC、对象分配过多、堆内存泄漏通常与堆有关，而 `StackOverflowError` 则常和递归或栈深度相关。

### 追问点
- 堆和栈的核心区别是什么？
- 哪个区域通常不会出现 `OutOfMemoryError`？
- 线上排查 OOM 时通常会看哪些信息？

### 评分点
- 至少能完整说出几个核心区域及其作用
- 能把内存区域和线上故障联系起来
- 回答不只停留在定义层面

### 常见失分点
- 把方法区、堆、栈混为一谈
- 只会背概念，不知道与排障的关系

## Q007 垃圾回收机制

### 元数据
- 岗位：Java后端开发工程师
- 题型：技术知识
- 主题：JVM
- 难度：高级
- 关键词：`垃圾回收` `GC` `分代` `G1`

### 面试题
你怎么理解 Java 的垃圾回收机制？线上如果 Full GC 很频繁，你会怎么排查？

### 标准回答
Java 垃圾回收的核心目标是自动识别和回收不再使用的对象，减少手动管理内存带来的风险。判断对象是否可回收，主流是通过可达性分析。堆内存通常会分为新生代和老年代，新生代对象存活率低、回收更频繁，老年代对象生命周期更长、回收成本更高。不同垃圾收集器会组合使用复制、标记清除、标记整理等算法。线上如果 Full GC 很频繁，通常要先看 GC 日志和监控，确认是对象晋升过快、内存泄漏、老年代空间不足还是大对象分配导致，再结合 `jmap`、`jstat`、heap dump 等工具分析对象分布和可疑引用链，最后再决定是优化代码、调整参数还是更换收集器。

### 追问点
- 哪些对象可能更容易晋升到老年代？
- CMS、G1、ZGC 的侧重点分别是什么？
- 哪些代码问题可能导致频繁 Full GC？

### 评分点
- 能讲清可达性分析和分代思路
- 能把 GC 机制和线上排查方法结合起来
- 能体现工程视角而不只是教材定义

### 常见失分点
- 只会背年轻代和老年代，不会排查问题
- 说 GC 问题只能靠调大内存解决

## Q008 MySQL 事务隔离级别

### 元数据
- 岗位：Java后端开发工程师
- 题型：技术知识
- 主题：MySQL
- 难度：中级
- 关键词：`MySQL` `事务` `隔离级别` `MVCC`

### 面试题
MySQL 事务的四种隔离级别分别是什么？各自解决了什么问题？

### 标准回答
MySQL 常见的四种事务隔离级别是读未提交、读已提交、可重复读和串行化。读未提交隔离性最低，会出现脏读、不可重复读和幻读。读已提交可以避免脏读，但仍可能出现不可重复读和幻读。可重复读进一步保证同一事务内多次读取同一数据结果一致，在 InnoDB 中配合 MVCC 和间隙锁还能在很多场景下减少幻读，是 MySQL 的默认隔离级别。串行化隔离性最高，会对读写都进行更严格的控制，并发性能也最差。回答时最好补充：实际工程中既要理解定义，也要知道隔离级别选择和锁竞争、吞吐量之间存在权衡。

### 追问点
- 什么是脏读、不可重复读、幻读？
- InnoDB 为什么默认用可重复读？
- MVCC 在这里起到了什么作用？

### 评分点
- 能准确说出四种隔离级别
- 能对应说明它们解决的问题
- 能提到工程上的隔离性与性能权衡

### 常见失分点
- 只会背顺序，不知道现象定义
- 把不可重复读和幻读混淆

## Q009 Redis 缓存一致性

### 元数据
- 岗位：Java后端开发工程师
- 题型：场景题
- 主题：Redis
- 难度：中高级
- 关键词：`Redis` `缓存一致性` `双删` `延迟双删`

### 面试题
在缓存和数据库同时存在的系统里，更新数据时你一般如何处理缓存一致性问题？

### 标准回答
常见思路是优先更新数据库，再删除缓存，而不是先更新缓存再更新数据库。因为缓存更适合做派生层，不应该作为最终真值来源。更新数据库后删除缓存，后续读请求在 cache miss 时再从数据库加载最新值，可以降低脏数据长期驻留的概率。如果业务一致性要求更高，还可以引入延迟双删、消息队列异步删除、订阅 binlog 同步缓存等方案。真正的工程回答要强调没有绝对完美的一致性方案，要根据读写比例、脏数据容忍度、延迟要求和复杂度成本做选择。

### 追问点
- 为什么一般不推荐先更新缓存再更新数据库？
- 什么场景下会使用延迟双删？
- 如果删除缓存失败了，通常怎么补偿？

### 评分点
- 能说出“更新数据库后删缓存”的主流思路
- 能说明方案并非绝对强一致
- 能给出失败补偿或增强方案

### 常见失分点
- 说缓存和数据库一定能做到实时绝对一致
- 只会背“双删”，不知道为什么这么做

## Q010 接口幂等设计

### 元数据
- 岗位：Java后端开发工程师
- 题型：场景题
- 主题：接口设计
- 难度：中高级
- 关键词：`幂等` `接口设计` `防重` `唯一键`

### 面试题
如何设计一个幂等接口，避免重复提交导致数据重复写入？

### 标准回答
幂等设计的核心是让同一请求重复执行多次，最终结果和执行一次保持一致。常见做法包括：前端提交唯一请求号，服务端基于请求号做去重；数据库层设置唯一索引，利用约束兜底；对于支付、下单等关键操作，可以在业务状态机中增加“处理中、成功、失败”等状态流转，避免重复创建资源；必要时配合分布式锁、Token 校验、去重表或 Redis 标记控制并发窗口。回答时要强调：幂等不是只靠某一个点解决，而是前端、服务端、数据库多层一起兜底。

### 追问点
- 唯一索引和请求号去重分别解决什么问题？
- 幂等和防重放攻击是不是一回事？
- 如果客户端超时重试，如何保证最终结果一致？

### 评分点
- 能讲出至少两层以上的幂等控制手段
- 能区分业务幂等和接口去重
- 能结合订单、支付等场景说明方案

### 常见失分点
- 把幂等简单理解成“接口不能重复调用”
- 只说加锁，不考虑重试和数据库兜底

## Q011 接口性能优化

如果一个核心接口响应时间从 200ms 上升到 2s，你会怎么排查和优化？

### 标准回答

### 追问点
- 如何快速判断问题是在应用层还是数据库层？
- 哪些场景适合异步化，哪些不适合？
- 优化后你会看哪些指标确认真的生效？

### 评分点
- 有明确的定位顺序，而不是只说“加缓存”
- 能覆盖应用、数据库、外部依赖多个层面
- 强调优化后的监控验证闭环

### 常见失分点

## Q012 项目深挖与行为协作

### 元数据
- 岗位：Java后端开发工程师
- 题型：项目深挖
- 主题：项目
- 难度：中级
- 关键词：`项目深挖` `指标` `复盘` `协作`

### 面试题
请你分享一个你负责推进并最终落地的后端项目，重点讲清楚背景、你的职责、技术难点、效果指标以及复盘。

### 标准回答

## Q096 如何处理线上与测试环境不一致

### 元数据
- 岗位：Java后端开发工程师
- 题型：场景题
- 主题：项目
- 难度：中级
- 关键词：`环境差异` `配置` `排查` `发布`

### 面试题
如果测试环境一切正常，但线上出现问题，你通常会怎么分析“环境不一致”带来的影响？

### 标准回答
这类问题通常要从配置、数据量、依赖版本、机器资源、流量特征、网络环境、开关状态和第三方服务差异几个方面排查。测试环境通过并不代表线上一定没问题，因为线上往往有更真实的数据分布、并发和依赖复杂度。工程上常见做法包括统一配置管理、缩小环境差异、做灰度验证、补压测和回放工具，从而降低“只在线上才暴露”的问题。

### 追问点
- 为什么数据量差异会影响 SQL 行为？
- 配置中心如何帮助降低环境差异？
- 流量回放有什么价值？

### 评分点
- 能从多维度看环境差异
- 能说明为什么测试通过不等于线上没问题
- 有治理思路而不只是排查

### 常见失分点
- 一律归因于“线上玄学”
- 不知道环境差异常见来源

## Q097 遇到不熟悉的问题怎么快速推进

### 元数据

- 难度：初级

如果你接到一个自己不熟悉的问题或模块，你通常怎么推进？

### 标准回答
面对不熟悉的问题，我通常会先明确目标和边界，再快速补上下文，例如看系统结构、接口调用链、最近变更、关键日志和相关文档。之后会把问题拆成几个更小的问题逐个验证，必要时请教熟悉该模块的同事，但会尽量带着自己的分析和假设去沟通，而不是空手提问。这样的做法能保证推进效率，也能体现独立解决问题的能力。

- 怎么判断先查文档还是先看代码？

### 评分点
- 有结构化推进思路
- 体现主动补上下文和拆解问题
- 不依赖“等别人告诉答案”

### 常见失分点
- 一上来就卡住不动
- 只会说“先百度一下”

## Q098 你怎么看代码可维护性

### 元数据
- 岗位：Java后端开发工程师
- 题型：行为题
- 主题：项目
- 难度：初级
- 关键词：`可维护性` `命名` `结构` `重构`

### 面试题
你觉得什么样的代码更有可维护性？

### 标准回答
可维护的代码通常具有清晰命名、合理分层、职责明确、边界清楚、异常处理完整、日志和监控到位、测试可验证等特点。它不一定是“最炫技”的代码，而是让后来的人容易理解、修改、排查和扩展。好的可维护性往往来自持续的小改进，而不是等到系统烂掉后一次性重构。

### 追问点
- 命名为什么这么重要？
- 什么叫职责单一？
- 可维护性和开发效率有冲突吗？

### 评分点
- 能从多个维度描述可维护性
- 有工程实践视角
- 不把可维护性只理解成代码风格

### 常见失分点
- 只会说“加注释”
- 不知道可维护性和排查效率关系

## Q099 如何和产品对齐技术风险

### 元数据
- 岗位：Java后端开发工程师
- 题型：行为题
- 主题：项目
- 难度：中级
- 关键词：`产品沟通` `风险对齐` `优先级` `预期管理`

### 面试题
如果技术实现存在明显风险，但产品希望快速上线，你会怎么沟通？

### 标准回答
这类沟通的关键不是直接拒绝，而是把风险说具体、说可感知、说可选择。通常会把可能的后果、影响范围、发生概率、是否有替代方案和最小可行路径讲清楚，让产品和业务方一起做决策。如果风险确实很高，也要明确表达底线并提供折中方案，例如拆阶段上线、缩小范围、先灰度验证。好的沟通既要有立场，也要有解决方案。

### 追问点
- 技术风险怎么讲才能让非技术同学理解？
- 什么情况下必须坚持底线？
- 如果最终仍然要上，你会做哪些兜底？

### 评分点
- 能体现风险表达和预期管理能力
- 有折中方案思维
- 不只是情绪化反对

### 常见失分点
- 只会说“不行”
- 讲不清风险到底是什么

## Q100 作为 Java 后端你最核心的能力是什么

### 元数据
- 岗位：Java后端开发工程师
- 题型：行为题
- 主题：项目
- 难度：中级
- 关键词：`核心能力` `工程思维` `学习能力` `稳定性`

### 面试题
你觉得一个 Java 后端工程师最核心的能力是什么？为什么？

### 标准回答
我认为核心能力不只是会写 Java 代码，而是能把业务问题转化成稳定、可维护、可演进的系统方案。这里面通常包括扎实的基础能力、排查问题的能力、性能和稳定性意识、协作沟通能力，以及持续学习和复盘能力。因为后端工程师面对的很多挑战都不是“API 会不会写”，而是系统在真实流量和复杂依赖下还能不能稳住、问题出了能不能快速定位、需求变化时能不能持续演进。

### 追问点
- 你觉得自己目前最强和最弱的点分别是什么？
- 你最近一次主动复盘和成长是什么？
- 如果让你继续提升，你会优先补哪一块？

### 评分点
- 回答有工程视角而不是泛泛而谈
- 能体现自我认知
- 能把技术能力和协作能力结合起来

### 常见失分点
- 只说“会很多框架”
- 回答空泛，没有自己的理解
