# Java后端补充知识库格式规范

> 适用对象：负责整理原理资料、经验文章、案例分析、排障总结的项目成员
> 适用范围：`D:\AI_Interview_Project\data\java_backend\` 下除主知识库外的补充资料
> 目标：保证补充库能为主库提供原理细节、工程案例和扩展说明，但不和主库角色混淆

---

## 1. 补充库定位

补充库不是主库的替代品。

补充库的职责是：

- 为主库答案补充技术细节
- 为追问提供更深的原理展开
- 为系统生成更像工程师的回答提供案例素材
- 为排障、优化、设计类问题提供经验背景

补充库不直接承担：

- 标准化出题
- 标准答案主模板
- 评分标准定义
- 面试失分点评估

一句话理解：

- 主库回答“这题怎么问、怎么答、怎么评”
- 补充库回答“这个知识点还能补充什么细节和案例”

---

## 2. 适合放进补充库的内容

以下内容适合放进补充库：

- Java 基础原理总结
- 集合源码解析
- 并发机制说明
- JVM 内存与 GC 长文
- MySQL 索引、事务、SQL 优化经验
- Redis 使用注意点
- 接口设计、安全设计、幂等、限流经验
- 线上问题排查案例
- 性能优化实践复盘
- 死锁、缓存一致性、线程池等专题文章

以下内容不适合放进补充库：

- 只有题目没有答案的题单
- 只含广告和外链的搬运文
- 与 Java 后端岗位无关的文章
- 前端、产品、设计类材料
- 过于零散且不可检索的随笔

---

## 3. 文件组织建议

补充库可以是多文件目录，不需要像主库一样强制单文件。

建议仍统一放在：

- `D:\AI_Interview_Project\data\java_backend\`

推荐命名方式：

- `Java基础-对象与字符串补充.md`
- `集合-HashMap与ConcurrentHashMap补充.md`
- `并发-volatile与线程池补充.md`
- `JVM-GC与类加载补充.md`
- `MySQL-事务与索引补充.md`
- `Redis-缓存一致性与分布式锁补充.md`
- `接口设计-幂等安全限流补充.md`
- `性能优化-慢接口与慢SQL案例补充.md`

要求：

- 文件名尽量体现主题
- 一个文件聚焦一个主题域
- 不要一个文件混太多完全无关的主题

---

## 4. 推荐文件头格式

补充库文件建议保留 YAML Front Matter，方便说明文章定位。

推荐格式：

```md
---
title: 集合-HashMap与ConcurrentHashMap补充
role: Java后端开发工程师
knowledge_base_type: support
topic: collection
description: 用于补充 HashMap、ConcurrentHashMap、扩容、并发安全等面试知识点。
---
```

推荐字段：

- `title`
- `role`
- `knowledge_base_type`
- `topic`
- `description`

要求：

- `knowledge_base_type` 固定写 `support`
- `topic` 尽量和主库主题体系对齐

---

## 5. 内容结构规范

补充库不要求像主库那样每题都固定六小节，但仍建议尽量结构化。

推荐使用以下结构：

```md
# HashMap 与 ConcurrentHashMap 补充

## 适用知识点
- HashMap
- ConcurrentHashMap
- 扩容
- 哈希冲突

## 核心结论
这里先写结论摘要。

## 原理说明
这里写底层原理。

## 工程实践
这里写真实开发中的选型和注意事项。

## 常见误区
- 误区 1
- 误区 2

## 面试可扩展点
- 可用于追问的点 1
- 可用于追问的点 2
```

核心建议：

- 先写结论，再写展开
- 让模型容易提取“重点”
- 少写无关铺垫

---

## 6. 推荐小节

补充库文章建议优先使用以下小节中的若干个：

- `适用知识点`
- `核心结论`
- `原理说明`
- `工程实践`
- `案例分析`
- `排查思路`
- `常见误区`
- `面试可扩展点`

不要求每篇都全有，但建议至少具备：

- 核心结论
- 原理说明

如果是案例型文章，建议额外加：

- 案例分析
- 排查思路
- 复盘总结

---

## 7. 写作要求

### 7.1 结论优先

不要一上来先铺很长背景。

推荐顺序：

1. 先给结论
2. 再解释原理
3. 再补工程实践
4. 最后补误区和延展

### 7.2 面向检索

内容应尽量包含明确术语，例如：

- `HashMap`
- `ConcurrentHashMap`
- `volatile`
- `AQS`
- `MVCC`
- `间隙锁`
- `Redis 分布式锁`

这样更利于向量检索和关键词定位。

### 7.3 工程化表达

补充库不是教材摘录，也不是口水文。

推荐多写：

- 实际项目里怎么用
- 什么场景容易踩坑
- 为什么某种做法不推荐
- 怎么排查
- 怎么权衡

---

## 8. 主题标签建议

为了和主库对齐，补充库建议尽量围绕这些主题：

- `Java基础`
- `集合`
- `并发`
- `JVM`
- `MySQL`
- `Redis`
- `接口设计`
- `性能优化`
- `项目`

要求：

- 新文章尽量能映射到其中一个主题
- 主题不要无限细分

---

## 9. 补充库与主库的边界

下面这个判断方法很实用。

如果一份内容主要回答：

- “面试官会怎么问？”
- “候选人应该怎么答？”
- “怎么打分？”

它更适合放主库。

如果一份内容主要回答：

- “这个原理还能怎么展开？”
- “真实项目里怎么落地？”
- “这个坑是怎么踩出来的？”
- “线上故障怎么排查？”

它更适合放补充库。

---

## 10. 禁止内容

补充库中不建议出现以下内容：

- 大段广告宣传语
- 公众号引流文案
- 和文章无关的二维码说明
- 大段 HTML/CSS 样式残留
- 纯图片、无文字解释的文档
- 完全未清洗的网页复制文本
- 乱码文本

这些内容会明显降低检索质量。

---

## 11. 清洗建议

把原始文章转成补充库材料时，建议做这些清洗：

1. 删除广告和外链导流内容
2. 删除与主题无关的开场寒暄
3. 保留真正有价值的原理、案例、结论
4. 给文章补上清晰标题
5. 用二级、三级标题重新组织结构
6. 尽量统一为 UTF-8 编码

---

## 12. 入库建议

补充库建议单独建 collection，不要和主库混在一个 collection 里。

推荐命名：

- `java_interview_support`

推荐入库命令：

```powershell
D:\AI_Interview_Project\scripts\.venv\Scripts\python.exe D:\AI_Interview_Project\scripts\ingest_data.py `
  --data-dir D:\AI_Interview_Project\data `
  --db-dir D:\AI_Interview_Project\db_support `
  --collection-name java_interview_support `
  --roles java_backend `
  --embedding-model D:\AI_Interview_Project\models\bge-small-zh-v1.5-full `
  --local-model-only `
  --clear-collection
```

注意：

- 这个命令适合在 `java_backend` 中放的是“纯补充库文件”时使用
- 如果目录里同时放了主库和补充库，建议后续改为 `--files` 精准指定

---

## 13. 推荐使用方式

补充库更适合这样用：

1. 先查主库，找到最相关题目
2. 再查补充库，拿到细节解释和工程案例
3. 最终回答时以主库为骨架，以补充库为扩展

这样做的好处是：

- 回答不跑题
- 面试结构更稳定
- 细节更丰富
- 答辩时更容易解释系统设计层次

---

## 14. 最小合格标准

一篇可以入补充库的文档，至少满足：

- 有清晰标题
- 有明确主题
- 有可检索的技术术语
- 有原理说明或工程实践价值
- 没有明显乱码和广告污染
- 内容与 Java 后端岗位面试相关

如果只是杂乱原文堆积，建议先清洗再入库。
