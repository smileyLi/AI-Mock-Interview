## 题目1

### 元数据
- 岗位：Java后端开发
- 题型：技术知识
- 主题：Java基础
- 难度：初级
- 关键词：`equals` `==` `hashCode` `对象比较`

### 面试题
请你解释一下 `==` 和 `equals()` 的区别？`hashCode()` 和它们之间有什么关系？

### 标准回答
`==` 用于比较两个变量的值是否相等。对于基本类型来说，比的是数值本身；对于引用类型来说，比的是两个引用是否指向同一个对象。`equals()` 是 `Object` 的方法，默认实现本质上也是比较地址，但很多类会重写它，比如 `String`，用于比较内容是否相等。`hashCode()` 通常和 `equals()` 配合使用，规则是：如果两个对象 `equals()` 为 `true`，那么它们的 `hashCode()` 必须相同；反过来 `hashCode()` 相同不一定 `equals()` 相同。这个规则在 `HashMap`、`HashSet` 等基于哈希的容器里非常重要。

### 追问点
- 为什么重写 `equals()` 时通常也要重写 `hashCode()`？
- 如果只重写 `equals()` 不重写 `hashCode()`，会出现什么问题？
- `String` 为什么适合作为 `HashMap` 的 key？

### 评分点
- 能区分基本类型和引用类型下 `==` 的含义
- 能说明 `equals()` 的默认行为和重写行为
- 能讲清楚 `hashCode()` 与哈希容器的关系

### 常见失分点
- 说 `==` 永远比较地址
- 说 `hashCode()` 相同就一定是同一个对象

## 题目2

### 元数据
- 岗位：Java后端开发
- 题型：技术知识
- 主题：集合
- 难度：中级
- 关键词：`ArrayList` `LinkedList` `随机访问` `插入删除`

### 面试题
`ArrayList` 和 `LinkedList` 有什么区别？分别适合什么场景？

### 标准回答
`ArrayList` 底层是动态数组，查询效率高，支持快速随机访问，但中间位置插入和删除元素时可能需要搬移数据。`LinkedList` 底层是链表，理论上插入和删除节点更方便，但随机访问性能差，查找某个位置元素需要遍历。实际开发中，如果主要是读操作、按下标访问较多，一般优先用 `ArrayList`；如果频繁在头尾插入删除，并且不依赖随机访问，可以考虑 `LinkedList`。不过现代业务开发里，`ArrayList` 使用更普遍。

### 追问点
- 为什么说 `LinkedList` 理论插入快，但实际业务里不一定更快？
- `ArrayList` 扩容机制是什么？
- `ArrayList` 和数组有什么区别？

### 评分点
- 能正确描述底层数据结构
- 能结合读写特点说明适用场景
- 能意识到实际性能不只看理论复杂度

## 题目3

### 元数据
- 岗位：Java后端开发
- 题型：技术知识
- 主题：集合
- 难度：中级
- 关键词：`HashMap` `哈希冲突` `扩容` `JDK8`

### 面试题
请你讲一下 `HashMap` 的底层实现原理。

### 标准回答
`HashMap` 是基于数组加链表加红黑树实现的哈希表。插入元素时，先根据 key 的 `hash` 计算桶下标，如果该位置为空就直接放入；如果发生哈希冲突，就在该桶里继续比较 key 是否相同，相同则覆盖，不同则挂到链表或红黑树中。JDK8 之后，当链表长度超过阈值且数组容量足够时，会把链表转成红黑树，以优化极端情况下的查找性能。`HashMap` 还会根据负载因子触发扩容，扩容后会重新分布元素位置。

### 追问点
- 为什么容量通常设计成 2 的幂？
- 什么时候链表会树化？
- `HashMap` 为什么线程不安全？

### 评分点
- 能说出数组+链表+红黑树结构
- 能说清碰撞处理和扩容
- 能提到 JDK8 的树化优化

### 常见失分点
- 只会背“数组加链表”，漏掉红黑树
- 说 `HashMap` 默认线程安全

## 题目4

### 元数据
- 岗位：Java后端开发
- 题型：技术知识
- 主题：并发
- 难度：中级
- 关键词：`线程池` `corePoolSize` `maximumPoolSize` `拒绝策略`

### 面试题
为什么生产环境通常不建议直接使用 `Executors` 创建线程池？你一般怎么配置线程池？

### 标准回答
生产环境不建议直接使用 `Executors` 的一些快捷工厂方法，主要是因为它们可能隐藏了不合理的队列或线程数量设置。例如固定线程池可能搭配无界队列，导致任务堆积和内存风险；缓存线程池可能创建大量线程，导致系统资源耗尽。更常见的做法是直接使用 `ThreadPoolExecutor`，显式指定核心线程数、最大线程数、队列容量、线程工厂和拒绝策略，并结合业务类型设置参数，比如 CPU 密集型和 IO 密集型的配置就不一样。

### 追问点
- 线程池的执行流程是什么？
- 常见拒绝策略有哪些？
- 任务异常在线程池里怎么处理？

### 评分点
- 能指出 `Executors` 的风险点
- 能说出 `ThreadPoolExecutor` 的核心参数
- 能结合业务场景谈配置思路

## 题目5

- 主题：并发
- 难度：中高级
- 关键词：`volatile` `可见性` `有序性` `原子性`

### 面试题
`volatile` 关键字有什么作用？能保证线程安全吗？

### 标准回答
`volatile` 主要保证变量的可见性和一定程度上的有序性。一个线程修改了被 `volatile` 修饰的变量，其他线程能够及时看到最新值；同时它还能禁止特定类型的指令重排序。但 `volatile` 不能保证复合操作的原子性，例如 `i++` 仍然不是线程安全的，因为它包含读、改、写多个步骤。如果需要对复合操作进行并发控制，通常还需要使用 `synchronized`、`Lock` 或原子类。

### 追问点
- `volatile` 和 `synchronized` 的区别是什么？
- 双重检查单例为什么要配合 `volatile`？
- `i++` 为什么不是原子操作？

### 评分点
- 说清可见性和有序性
- 明确指出不保证复合操作原子性
- 能举出实际使用场景

## 题目6

### 元数据
- 岗位：Java后端开发
- 题型：技术知识
- 主题：JVM
- 难度：中高级
- 关键词：`JVM` `内存模型` `堆` `栈` `方法区`

### 面试题
请你讲一下 JVM 运行时内存区域。

### 标准回答
JVM 运行时内存区域通常包括程序计数器、虚拟机栈、本地方法栈、堆和方法区。程序计数器是线程私有的，用来记录当前执行位置；虚拟机栈保存方法调用过程中的栈帧信息；本地方法栈服务于 native 方法；堆是对象实例和数组的主要分配区域，也是垃圾回收的重点；方法区用于存储类元信息、常量、静态变量等。排查线上问题时，像 `OutOfMemoryError`、频繁 GC、对象创建过多等都和这些区域密切相关。

### 追问点
- 堆和栈的区别是什么？
- 常见的 OOM 场景有哪些？
- 为什么程序计数器一般不会 OOM？

### 评分点
- 至少能说出几个核心区域及作用
- 能把内存区域和排错联系起来
- 回答不要只停留在背概念

## 题目7

### 元数据
- 岗位：Java后端开发
- 题型：技术知识
- 主题：JVM
- 难度：高级
- 关键词：`垃圾回收` `GC` `年轻代` `老年代`

### 面试题
你怎么理解 Java 的垃圾回收机制？

### 标准回答
Java 垃圾回收的核心目标是自动识别和回收不再被引用的对象，减少手动管理内存带来的风险。常见判断对象是否可回收的方式是可达性分析。堆内存通常按代划分为年轻代和老年代，年轻代对象朝生夕灭较多，因此回收更频繁；老年代对象生命周期更长，回收成本更高。不同垃圾收集器会采用标记清除、复制、标记整理等算法组合，实际线上关注点通常包括 GC 次数、停顿时间、吞吐量和内存占用。

### 追问点
- 什么对象会晋升到老年代？
- CMS 和 G1 有什么区别？
- Full GC 频繁通常怎么排查？

### 评分点
- 能说出可达性分析
- 能理解分代回收思想
- 能从工程角度谈 GC 指标和排查

### 面试题
MySQL 索引有什么作用？什么时候索引会失效？

### 标准回答
索引的作用是加快数据检索，减少全表扫描，提高查询效率。常见的索引结构是 B+ 树，它适合范围查询和排序。索引并不是越多越好，因为索引也会占用空间，并增加写入成本。索引失效的常见场景包括：没有遵循最左前缀原则、对索引列进行了函数计算或隐式类型转换、使用了不合适的模糊查询方式、选择性太差导致优化器放弃索引等。

### 追问点
- 什么是覆盖索引和回表？
- 联合索引的最左匹配原则是什么？
- 为什么 `like '%xx'` 容易失效？

### 评分点
- 能说出索引提升查询效率但增加写成本
- 能举出多个失效场景
- 能结合 SQL 调优说明影响

## 题目9

### 元数据
- 岗位：Java后端开发
- 题型：技术知识
- 主题：数据库
- 难度：中高级
- 关键词：`事务` `隔离级别` `脏读` `幻读`

### 面试题
MySQL 事务的四种隔离级别是什么？分别解决了什么问题？

### 标准回答
事务隔离级别从低到高一般是：读未提交、读已提交、可重复读、串行化。读未提交可能发生脏读、不可重复读和幻读；读已提交解决了脏读，但仍可能有不可重复读和幻读；可重复读解决了脏读和不可重复读，在 MySQL InnoDB 中通过 MVCC 和间隙锁等机制对幻读也做了较强控制；串行化隔离性最强，但并发性能最差。实际业务中常见默认级别是可重复读。

### 追问点
- 什么是 MVCC？
- 脏读、不可重复读、幻读分别是什么意思？
- 高并发订单场景下如何权衡隔离性和性能？

### 评分点
- 能完整说出四种隔离级别
- 能对应三类典型并发问题
- 能理解隔离级别和性能的权衡

## 题目10

### 元数据
- 岗位：Java后端开发
- 题型：技术知识
- 主题：缓存
- 难度：中高级
- 关键词：`Redis` `缓存穿透` `缓存击穿` `缓存雪崩`

### 面试题
缓存穿透、缓存击穿、缓存雪崩分别是什么？怎么解决？

### 标准回答
缓存穿透是请求查询一个缓存和数据库里都不存在的数据，导致请求直接打到数据库，常见解决方案有布隆过滤器和缓存空值。缓存击穿通常是某个热点 key 在失效瞬间，大量并发请求同时落到数据库，可通过热点数据永不过期、互斥锁、逻辑过期等方案缓解。缓存雪崩是大量 key 在同一时间失效，导致后端压力激增，常见方案是过期时间加随机值、分批失效、降级限流和多级缓存。

### 追问点
- 布隆过滤器适合什么场景？
- 热点 key 怎么识别？
- 为什么缓存空值也要设置过期时间？

### 评分点
- 能区分三者定义
- 能给出对应治理方案
- 能体现工程上的防护思路

- 能体现责任感和复盘能力
- 不甩锅，能讲清后续改进

## 统一评估建议

### 技术回答评估维度
- 正确性：概念是否准确，有无明显事实错误
- 深度：是否能从原理讲到工程实践
- 结构性：回答是否有层次，而不是散点堆砌
- 岗位匹配度：是否贴近 Java 后端实际场景

### 表达评估维度
- 是否先给结论再展开
- 是否能用例子辅助说明
- 是否能在追问下保持逻辑稳定

### 生成反馈建议
- 对基础题：指出概念漏洞和术语误用
- 对场景题：指出方案完整性、风险意识和 trade-off
- 对行为题：指出表达是否有“背景-行动-结果-复盘”
